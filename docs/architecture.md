# Architecture

The app is split so the live drafting UI can later move to React Native, Flutter, Capacitor, or a desktop shell without rewriting the draft logic.

## Frontend

`index.html`, `src/app.js`, and `src/styles.css` render the web client.

The client owns fast interaction state only:

- selected draft philosophy
- selected user role
- visible ally and enemy picks
- final hero screen state

## Recommendation Engine

`src/engine/recommendation-engine.js` is pure domain logic. It can run in:

- browser
- Node API service
- serverless functions
- mobile backend
- desktop backend

It has no DOM dependency.

## Data Layer

`src/draft-data.js` contains the structured Patch 7.41c seed dataset.

`data/dotabuff-counters-7.41c.json` is the intended generated database output from the scraping system.

`data/hero-database-7.41c.json` is the intended full hero database generated by `scripts/full-hero-database-ingest.mjs`. It is designed to include every hero returned by OpenDota/Valve data, so it will handle 127, 128, or any future roster size without hardcoding the number.

## Scraping System

`scripts/dotabuff-ingest.mjs` extracts hero counter pages into structured records.

`scripts/full-hero-database-ingest.mjs` builds the complete per-hero dataset:

- OpenDota hero list and hero IDs
- OpenDota item popularity by hero
- OpenDota matchup data by hero
- Dotabuff overview page per hero
- Dotabuff items page per hero
- Dotabuff counters page per hero

Dotabuff can reject scripted access with `403 Forbidden`; the ingester records those failures instead of crashing.

Future production ingestion should run as a scheduled backend job and combine:

- Dotabuff counters
- OpenDota matchups
- Dota2ProTracker item and role trends
- curated high-rank/pro draft annotations

## API Service

`backend/server.mjs` exposes:

- `POST /api/recommendations`
- `POST /api/final`

`api/recommendations.mjs` is a Vercel-style serverless entry point.

## Deployment

Static web hosting:

- Vercel
- Netlify
- Cloudflare Pages

Backend hosting:

- Railway
- Render
- Supabase Edge Functions
- AWS
- DigitalOcean

Mobile future:

- keep `src/engine` and `src/services` shared
- replace only the UI shell
- call the same API service from mobile clients
